Sample deliverable · Product-update enablement

One update, three formats, one day

When a product ships fast, enablement’s job is to turn each change into the right learning for each audience before it shows up as a wave of tickets. This sample takes a single update and shows the same information rendered three ways — a team blast, a specialist quick-update, and a knowledge-base article — plus how I’d decide what goes where.

About this sample

Independent work sample built from Harvey’s public product information; not affiliated with or endorsed by Harvey, and containing no confidential material. The “update” below is framed as an illustrative example based on a publicly described capability — review tables available directly within Assistant — used here only to demonstrate format and process.

The input

The release, in one line

Illustrative update: users can now generate a review table — structured, comparable insights across many documents — directly from within Assistant, instead of only inside a Vault. That’s the raw change. Everything below is the enablement response.

Format 1

Team blast — awareness in 20 seconds

📢 Shipping today: review tables inside Assistant

Users can now build a review table directly in Assistant — not just in a Vault. Expect questions like “where did the table feature go / why is it here now” and “how is this different from a Vault review table.” Full quick-update and a KB article are in the enablement hub. TL;DR: same capability, new place to start it. Verify-the-citations guidance is unchanged.

Format 2

Specialist quick-update — enough to handle the tickets

1 What changed

Review tables — extracting and comparing key data points across many documents in a structured grid — can now be created from within Assistant, in addition to within a Vault.

2 Who’ll ask, and what to say

“Is this new or did it move?” — It’s an additional entry point; the Vault experience still exists.

“Which should I use?” — Assistant for a quick, in-the-flow comparison; a Vault when the document set is large or shared and you want it stored and reusable.

Unchanged: outputs are still cited — coach users to verify each cell against its source before relying on it.

3 If something looks off

Same triage as any extraction: confirm source scope first, then check the citation. Route substantive accuracy issues through the normal escalation path and tag them so we can see whether the new entry point changes ticket patterns.

Format 3

Knowledge-base article — the durable, linkable answer

Creating a review table from Assistant

What it is: a review table extracts and compares key data points across multiple documents in a structured, tabular view. You can now start one directly in Assistant.

When to use which: use Assistant for a fast comparison in the middle of your work; use a Vault when you’re working over a large or shared document set you want stored, permissioned, and reusable across the platform.

Good to know: results are cited. Expand any cell’s citation to confirm it against the source before you rely on it — especially for dates, figures, and anything outcome-determinative.

If a result looks wrong: first confirm the document was in scope for the query; then open the citation to check the source. If the source genuinely doesn’t support the cell, contact support with the query and the cited passage.

Closing the loop

How I’d know it landed

Content isn’t done when it ships — it’s done when it changes something. My primary metric for an update like this is the topic ticket rate: tickets about review tables per 100 active users, tracked week over week. If awareness landed, it shouldn’t spike when the feature rolls out; if it climbs, the blast or the article missed. Around it I’d watch two supporting signals — the KB article’s “was this helpful?” rate (is the content itself clear?) and whether tagged accuracy tickets shift after the new entry point (a new failure mode to retrain on?).

One distinction I care about: ticket rate is a lagging signal, so I pair it with a leading one — specialist confidence on the topic, sampled in a 30-second pulse check — to catch confusion before it reaches the queue. Those signals decide what to revise, which is the start of the next update, not the end of this one.
